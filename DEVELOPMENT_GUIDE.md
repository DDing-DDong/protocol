## 👥 5인 팀 역할분담 (확정)

### 팀원 A: 해커 조작 담당 (공격 턴 개발)

**책임:**
- 해커 캐릭터의 모든 움직임 (이동, 점프, 대시, 방어막)
- 에너지 시스템 관리
- 피해 처리 로직

**주의사항:**
- 조작감(느낌)이 가장 중요 → 매일 테스트하고 느낌이 자연스러운지 확인
- 점프 높이, 대시 거리를 게임 밸런싱과 협력해서 조정
- 현재 코드: `updateAttack()`, `createHacker()`, `moveAndCollide()`

**매일 체크리스트:**
- [ ] 해커가 자연스럽게 움직이는가?
- [ ] 대시의 거리와 쿨타임이 적절한가?
- [ ] 에너지 소모가 게임 진행 속도와 맞는가?

---

### 팀원 B: 리플레이 시스템 담당 ⭐ **가장 중요**

**책임:**
- 공격 턴 경로 기록 정확도
- 수비 턴 리플레이 재생 정확도
- 고스트 해커 표시 (시각화)

**현재 문제점:**
- 0.07초 간격 샘플링 방식이 불안정할 수 있음
- 물리 엔진 동기화 필요

**해결 방안 (MVP):**
- 프레임 기반 기록 추가 고려 (입력 + 타임스탬프)
- 재생할 때 입력 재현식으로 변경 (더 안정적)

**현재 코드:**
- `recordHacker()` - 경로 기록
- `updateDefenseReplay()` - 리플레이 재생
- `createReplayHacker()` - 고스트 해커 생성

**매일 체크리스트:**
- [ ] 같은 경로를 10회 재생해도 동일한가?
- [ ] 리플레이 중 물리가 이상하지는 않은가?
- [ ] 고스트 표시가 시각적으로 명확한가?

**알아야 할 것:**
- A의 input을 정확히 따라야 함
- 시간 기반 vs 프레임 기반 샘플링의 차이 이해

**책임:**

## 🏗️ 시스템 아키텍처 (코드 이해도)

### 데이터 흐름

```
공격 턴:
A(해커 조작) → B(경로 기록) → game.currentRecording 저장
                             ↓
                        게이트웨이: game.lastAttackRecording에 복사

수비 턴:
D(스테이지 준비) → game.lastAttackRecording 로드
                 ↓
              C(함정 배치) → game.placedTraps 저장
                         ↓
                    B(리플레이) → game.replayHacker 이동
                                ↓
                             C(함정 효과) → 탐지/지연 계산
                                         ↓
                                      D(목표 판정) → 성공/실패 결정
                                                    ↓
                                                 E(보상 표시) → 보상 선택
                                                             ↓
                                                          A/D(다음 턴) 준비
```

### game 객체 구조

```javascript
{
  stage: 현재 스테이지,
  turn: "attack" | "defense_build" | "defense_replay" | "ending",
  
  // A(해커) 담당
  hacker: { x, y, vx, vy, hp, energy, ... },
  timer: 남은 시간,
  
  // B(리플레이) 담당
  currentRecording: [{t, x, y, facing, shield}],
  lastAttackRecording: [{t, x, y, facing, shield}],
  replayHacker: { x, y, facing, hp, ... },
  
  // C(함정) 담당
  placedTraps: [{id, type, x, y, slotId}],
  defenseBudget: 남은 예산,
  
  // D(스테이지) 담당
  platforms: [{x, y, w, h}],
  baseHazards: [{type, x, y, w, h}],
  trapSlots: [{x, y, id, occupied}],
  
  // 전체 공유
  metrics: { detections, delay, energyUsed, ... },
  mods: { maxEnergy, dashCooldown, ... },
}
```

## ⚠️ 개발 중 주의사항

### 가장 위험한 부분

#### 1️⃣ **B(리플레이)가 망가지면 게임이 망한다**
- 지연 시간 0.07초는 60fps 기준 약 4프레임
- 만약 네트워크 지연이나 버그가 있으면 경로가 흐트러짐
- **해결:** 프레임 기반 기록으로 변경 고려

#### 2️⃣ **C(함정)와 B(리플레이)의 충돌 판정이 정확해야 함**
- hitbox 계산이 틀리면 함정이 작동하지 않음
- 각 함정마다 hitbox 형태가 다름
- **해결:** 함정별 hitbox 테스트 케이스 작성

#### 3️⃣ **E(UI)가 못 따라가면 플레이어가 헷갈린다**
- 공격 턴에 함정 배치창이 보이면 안 됨
- 수비 턴에 에너지 바가 보여야 함
- **해결:** E는 매일 UI 요소 체크리스트 작성

#### 4️⃣ **D(스테이지)가 밸런스를 조정하지 못하면 게임이 너무 쉽거나 어렵다**
- 제한 시간, 함정 예산, 보상이 모두 연관되어 있음
- 수치를 1개만 바꿔도 게임 난이도가 확 바뀜
- **해결:** 밸런싱은 나중에 2-3회 수정 필요

## 🎨 코드 스타일 가이드

모든 팀원이 같은 규칙으로 코드를 작성해야 통합할 때 혼란이 없습니다.

```javascript
// ✅ 좋은 예시
function applyHazard(entity, hazard, dt) {
  if (!rectsOverlap(entity, hazard)) return; // 빠른 반환
  // 로직...
}

// ❌ 피할 것
function applyHazard(entity, hazard, dt) {
  if (rectsOverlap(entity, hazard)) {
    // 로직...
  }
}
```

**명명 규칙:**
- 변수: `camelCase` (예: `currentStage`, `replayIndex`)
- 상수: `UPPER_CASE` (예: `GRAVITY`, `GROUND_Y`)
- 함수: `camelCase` (예: `updateAttack()`, `rectsOverlap()`)
- 함정 타입: 소문자 (예: `"laser"`, `"shock"`)
